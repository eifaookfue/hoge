スライド案（そのまま貼れる文章）

タイトル案

ソースリポジトリとデプロイメントリポジトリの責務分離

⸻

左：ソースリポジトリ（Source Repository）

役割
	•	アプリケーションのソースコードを管理
	•	コンパイル、単体テスト、静的解析を実行
	•	Tag Push によりリリース成果物（JAR / コンテナイメージ）を作成

管理対象
	•	Java / React ソースコード
	•	build.gradle / pom.xml
	•	テストコード
	•	CI定義（GitHub Actions workflow）

成果物
	•	JARファイル
	•	コンテナイメージ（Tag＝Git Tag）

⸻

右：デプロイメントリポジトリ（Deployment Repository）

役割
	•	実行環境に必要な「デプロイ定義」を管理
	•	環境ごとの起動方法を統一
	•	CD実行時に compose を取得し、起動する

管理対象
	•	podman-compose.yml
	•	.env（環境変数定義）
	•	起動/停止スクリプト
	•	環境ごとの差分（dev / uat / prod）

成果物
	•	デプロイ可能な状態の起動定義（構成）

⸻

なぜ分けて管理するのか（下段に入れる）

分離するメリット
	•	ソースのライフサイクルと、デプロイ定義のライフサイクルが異なる
	•	ソース変更なしで環境設定だけ変えたいケースがある
	•	環境差分（dev/uat/prod）を安全に管理できる
	•	デプロイ方式の違い（Podman / Kubernetes等）に対応しやすい
	•	責務を明確化でき、アプリ開発者の負担を軽減できる
	•	変更履歴が残り、デプロイの再現性が高まる

⸻

注意点・補足（小さく右下に）
	•	デプロイ時に利用した Deployment Repo の commit SHA を記録し、追跡性を担保する
（例：デプロイマニフェスト、labels、ログ）

⸻

図として表現するなら（スライド構成案）

上段：2つの箱
	•	左箱：Source Repo
	•	右箱：Deployment Repo

中央：成果物の矢印

Source Repo →（CI）→ Artifact Repository / Container Registry（ECRなど）

下段：CDの矢印

Deployment Repo →（CD手動）→ podman-compose up → 実行環境

⸻

書き方のコツ（刺さる表現）

スライドでは「設定ファイル」って言うより、こう言うと説得力が上がります：
	•	設定ファイル ❌
	•	デプロイ定義 / 実行定義 / 環境定義 ✅

また、分離理由はこの一言が最強です：

「アプリケーションのソース」と「実行環境の定義」は変更理由が異なるため、リポジトリを分離する
